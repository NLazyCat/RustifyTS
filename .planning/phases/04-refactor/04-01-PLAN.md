# Phase 04: Semantic Refactoring Core Layer - Wave 1

**Plan ID:** 04-01
**Wave:** 1/3
**Estimated Time:** 2 days
**Dependencies:** Phase 03 complete (semantic analysis and IR generation)

## Goal

Build the foundational infrastructure for semantic refactoring: AnalysisContext, analyzer traits, and module structure. This provides the framework that RRA, OA, SR, and derivation algorithms will use.

## Context

From **04-CONTEXT.md** and **04-RESEARCH.md**, we have locked decisions:
- Analysis pipeline: RRA → OA → derivation (sequential)
- Result storage: Context results map, keyed by (NodeId, TypeId)
- Cache strategy: Lazy cache, invalidate on IR changes
- Incremental analysis: Full incremental, support updates on IR changes

From research, we must:
- Use hashbrown::HashMap for performance-critical collections
- Implement lattice-based dataflow analysis framework
- Support transformation tracking with history

## Must-Haves (Goal-Backward Verification)

To verify this wave succeeds, we MUST have:

1. **AnalysisContext structure** that:
   - Holds SemanticModule from Phase 03
   - Contains results map: HashMap<(NodeId, TypeId), Box<dyn AnalysisResult>>
   - Manages cache invalidation on IR changes
   - Supports incremental updates

2. **SemanticAnalyzer trait** that:
   - Defines analyze(&self, ctx: &mut AnalysisContext) -> Result<()>
   - Has name() method for identification
   - Supports R: AnalysisResult generic parameter
   - Clone + Debug + Send + Sync bounds on AnalysisResult

3. **Module structure** with:
   - src/semantic/refactor/mod.rs (public API)
   - src/semantic/refactor/context/ (AnalysisContext)
   - src/semantic/refactor/traits/ (analyzer interfaces)

4. **Cache invalidation mechanism** that:
   - Tracks IR version/hash
   - Invalidates results on changes
   - Supports incremental analysis flag

5. **Tests** covering:
   - AnalysisContext creation and mutation
   - Trait implementation for mock analyzers
   - Cache invalidation behavior
   - Results map CRUD operations

### Task 2: Implement AnalysisContext with Cache Invalidation

**Objective:** Build AnalysisContext that holds analysis results and manages cache invalidation.

**Steps:**
1. Create `src/semantic/refactor/context/mod.rs` with module exports
2. Create `src/semantic/refactor/context/context.rs` with:
   - `AnalysisContext` struct containing:
     - `module: SemanticModule` (from Phase 03)
     - `results: FxHashMap<(NodeId, TypeId), Box<dyn AnalysisResult>>`
     - `ir_version: u64` for change tracking
     - `cache_enabled: bool` flag
   - `impl AnalysisContext` methods:
     - `new(module: SemanticModule) -> Self`
     - `get_result<R: AnalysisResult>(&self, node: NodeId, ty: TypeId) -> Option<&R>`
     - `set_result<R: AnalysisResult>(&mut self, node: NodeId, ty: TypeId, result: R)`
     - `invalidate_cache(&mut self)` - clears results, increments version
     - `ir_version(&self) -> u64`
     - `increment_version(&mut self)` - called when IR changes
3. Implement `Clone` for AnalysisContext (deep copy results)
4. Add Debug impl for debugging

**Acceptance Criteria:**
- [ ] AnalysisContext compiles
- [ ] Results map uses FxHashMap for performance
- [ ] Cache invalidation clears results and increments version
- [ ] get_result/set_result work correctly
- [ ] Clone impl works
- [ ] Debug impl works

## Notes

- Use `FxHashMap` from fxhash crate (already in project) for performance
- Results map key is `(NodeId, TypeId)` tuple - this is the standard pattern
- Cache invalidation is lazy - only happens when explicitly called
- IR version is u64 to avoid overflow issues
- AnalysisResult is object-safe (dyn) to allow storing different result types
